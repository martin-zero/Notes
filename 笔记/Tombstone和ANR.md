**ANR** 和 **Tombstone** 是 Android 系统中两种最常见的**故障报告机制**，它们分别用于诊断不同类型和级别的系统失败。

简而言之：

- **Tombstone（墓碑文件）**：用于诊断 **原生代码（C/C++）的致命崩溃**，例如访问了无效的内存地址（SIGSEGV）。它记录了程序“死亡”时的状态。
    
- **ANR（Application Not Responding）**：用于诊断 **Java/应用层线程阻塞**，即应用长时间没有响应用户输入或系统广播。它记录的是程序“卡死”时的状态。
    

---

## 💀 Tombstone vs. ANR 对比表

|**特征**|**Tombstone (墓碑文件)**|**ANR (应用无响应)**|
|---|---|---|
|**故障类型**|**原生代码崩溃（Crash）**|**应用线程阻塞（Hanging/Stuck）**|
|**发生语言**|主要发生在 **C/C++** 等原生代码中。|主要发生在 **Java/Kotlin** 应用层代码中。|
|**触发信号**|接收到致命信号，如 `SIGSEGV`（段错误）、`SIGABRT`（中断）。|未响应系统事件（如用户输入、广播）超过**预设时间**。|
|**检测机制**|由内核和 `debuggerd` 服务捕获原生崩溃信号。|由 **AMS (Activity Manager Service)** 或 **WMS (Window Manager Service)** 持续监控应用的主线程活动。|
|**输出文件**|**Tombstone 文件**（`/data/tombstones/`），包含寄存器状态和原生堆栈。|**Trace 文件**（`/data/anr/traces.txt`），包含所有 Java 线程的堆栈信息。|
|**用户体验**|应用立即闪退，通常系统会提示“应用已停止运行”。|系统弹出对话框，提示用户“应用无响应，是否关闭？”。|
|**诊断重点**|查找**空指针解引用**、内存越界、双重释放等**内存安全问题**。|查找**死锁**、**主线程上的耗时 I/O 操作**或**死循环**等线程阻塞问题。|

---

## 详细解释

### 1. ANR (Application Not Responding)

**ANR** 是 Android 系统的自我保护机制。如果应用的主线程（UI 线程）在一段时间内（通常是 **5 秒**）没有处理完输入事件、广播或服务调用，系统就会认为该应用“卡死”了，即**无响应**。

- **核心：** 线程阻塞。应用本身没有崩溃，但它不能及时响应用户或系统。
    
- **后果：** 系统会显示 ANR 对话框，用户可以选择等待或强制关闭应用。
    
- **报告：** 系统会在 `/data/anr/` 目录下生成 **`traces.txt` 文件**。该文件记录了系统内所有进程中所有 Java 线程的完整堆栈信息，帮助开发者定位哪个线程卡住了主线程。
    

### 2. Tombstone（墓碑文件）

**Tombstone** 是一种针对**原生代码崩溃**的诊断文件。

- **核心：** 原生代码访问了非法内存。进程直接被内核终止。
    
- **后果：** 进程立即终止并退出。
    
- **报告：** 系统生成 **Tombstone 文件**，其中包含了崩溃瞬间的 CPU 寄存器状态、内存映射以及**原生堆栈轨迹**。这是解决底层 C/C++ 崩溃（如 `mediaserver` 进程崩溃）的唯一有效工具。
    

> **总结：** 如果应用是**卡住不动**然后弹出提示，那是 **ANR**。如果应用是**突然退出**且没有应用层的友好提示，那很可能是 **Tombstone** 导致的底层原生崩溃。
