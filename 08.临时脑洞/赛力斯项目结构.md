## IPC通信
（ 模块之间也用IPC通信）
**Proxy端** 客户端
继承MsgProxy
Connect建立连接
RegNotify注册广播
StartNotify通知广播订阅

**Server端** 服务端
继承MsgServer
Notify发送广播

⏺ 项目 Middleware 和 HMI 架构讲解

  一、项目概览

  这是一个 汽车仪表盘系统，运行在 QNX 实时操作系统上，基于 Kanzi 引擎渲染，面向 大众 Amarok
  车型。整个系统分为三层架构：

  Middleware Layer (中间件层)  →  HMI Data Adapter Layer (数据适配层)  →  HMI Rendering Layer (渲染层)

  ---
  二、Middleware 层 — 核心业务逻辑

  Middleware 编译为两个共享库：libClusterMiddleware.so（服务端逻辑）和 libClusterProxy.so（代理/通信接口）。

  模块间通过 FDBus（IPC 总线）通信，采用 Server/Proxy 模式：
  - XxxServer 继承 MsgServer，负责处理数据并发布通知
  - XxxProxy 继承 MsgProxy，负责发送命令/调用

  15 个模块：

  模块: SPIComm
  路径: Middleware/SPIComm/
  职责: SPI 通信层，与 MCU 微控制器交换 18 种数据帧（电源、转速、车速、水温、燃油、ODO、档位、告警、灯光等）。带 CRC16
    校验，支持离线缓存
  ────────────────────────────────────────
  模块: VehicleComm
  路径: Middleware/VehicleComm/
  职责: CAN 车辆信号接入，通过 Bosch 的 VehicleToolProto protobuf 协议接收 CAN 报文，支持 int32/float32 两种数据
  ────────────────────────────────────────
  模块: GaugeServer
  路径: Middleware/GaugeServer/
  职责: 仪表盘核心（6 个表盘）：水温表、车速表、转速表、燃油表、档位显示、RTC
  时间。含阻尼算法（平滑指针运动）和扫表特效
  ────────────────────────────────────────
  模块: PowerServer
  路径: Middleware/PowerServer/
  职责: 电源/生命周期管理。IGN ON/OFF 状态切换，LCM 挂起/恢复（Bosch SUS/DIF
  协议），自检流程（AutoCheck），息屏动画计时
  ────────────────────────────────────────
  模块: TelltaleServer
  路径: Middleware/TelltaleServer/
  职责: 69 个警示灯管理（ABS、EPS、安全气囊、安全带、车门等）。支持 11 种颜色状态。自检时控制各灯的亮灭序列，区分
    PHEV/柴油/EV 车型
  ────────────────────────────────────────
  模块: WarningServer
  路径: Middleware/WarningServer/
  职责: 告警弹出管理（267 条 SPI→Kanzi ID 映射表）。ADAS 告警优先显示在中间，普通告警在左侧。左侧 AVM
  开启时抑制左侧告警
  ────────────────────────────────────────
  模块: TCServer
  路径: Middleware/TCServer/
  职责: 行车电脑：里程表、小计里程、续航里程（柴油 ≤1999，PHEV
    ≤999）、瞬时油耗、平均油耗/车速、保养信息。含英制/公制单位换算
  ────────────────────────────────────────
  模块: SettingServer
  路径: Middleware/SettingServer/
  职责: 车辆设置持久化，通过 PSIS 存储 22 个 DID 值和 17 种用户设置（语言 26 种、主题 4 种、壁纸 6 种、地图类型 3
  种等）
  ────────────────────────────────────────
  模块: HMIStateServer
  路径: Middleware/HMIStateServer/
  职责: HMI 页面/主题管理。5 个页面（距离/消耗/水温/SCR/Gauge），4
    个主题（Classic/Gauge/Map/4th）。左右按钮翻页，上下切换主题，OK 确认。PHEV 跳过 SCR 页面
  ────────────────────────────────────────
  模块: ClusterIVIServer
  路径: Middleware/ClusterIVIServer/
  职责: 仪表与中控 IVI 的桥接，管理 AVM 全景影像状态、导航地图/窗口错误信息、行程重置等
  ────────────────────────────────────────
  模块: HotkeyServer
  路径: Middleware/Hotkey/
  职责: 方向盘按键处理（上下左右 OK），从 CAN 信号 HW_KEY_INPUT 接收，过滤长按
  ────────────────────────────────────────
  模块: Timer
  路径: Middleware/Timer/
  职责: 系统定时器，提供 10ms/20ms/50ms/100ms/1s 等定时，使用 CLOCK_MONOTONIC，支持 STR 恢复时检测时间跳变
  ────────────────────────────────────────
  模块: LcmProxy
  路径: Middleware/PowerServer/src/LcmProxy.cpp
  职责: LCM 生命周期代理，处理 SUS/DIF 协议：注册/取消、挂起确认、恢复确认、关机准备、背光亮度
  ────────────────────────────────────────
  模块: Logger
  路径: Middleware/Logger/
  职责: slog2 日志框架，16 个子系统分类，含启动性能计时
  ────────────────────────────────────────
  模块: AppWatchdog
  路径: Middleware/AppWatchdog/
  职责: 基于 QNX HAM 的进程守护，监控 cluster-app、cluster-fusa、cluster-hmi 三个进程，崩溃后自动重启

  数据流向：
  MCU ←→[SPI]→ SPICommServer →→→→→→→→→→→→→→→→→→→→→→→→→→→ GaugeServer, TCServer, TelltaleServer
  CAN ←→[Bosch Proto]→ VehicleCommProxy →→→→→→→→→→→→→→→→→→   WarningServer, HotkeyServer, SettingServer
  IVI ←→[FDBus]→ ClusterIVIServer →→→→→→→→→→→→→→→→→→→→→→→→ HMIStateServer, PowerServer
                                                                        |
                                                                        ↓
                                                                 HMI (Kanzi + FUSA 渲染)

  ---
  三、HMI 层 — 图形渲染

  HMI 层由两个并行的渲染进程组成：

  3.1 cluster-hmi（主仪表，Kanzi 引擎）

  - 路径： HMI/kanzi/ClusterFunction/
  - 渲染： Kanzi Engine + OpenGL ES 3.1，1920×720，30FPS，窗口组 DPY_CLUSTER_FUNCTION
  - 内容： 仪表盘（速度/转速/燃油/水温）、ADAS 叠加（车道线、障碍物、BSD盲区）、菜单页面、主题切换
  - 资产： 编译后的 KZB 文件（clusterfunction.kzb、adas.kzb、alarm.kzb、charging.kzb、resource.kzb）

  3.2 cluster-fusa（功能安全，QNX screen 直接合成）

  - 路径： HMI/kanzi/ClusterFusa/
  - 渲染： QNX screen API 直接像素合成，1920×720，60FPS，z-order 6
  - 内容： 功能安全相关元素——69 个警示灯图标、档位指示、弹出图标，绕过 Kanzi 确保可靠性
  - 图标： 预渲染的 IconMapping（384×864 RGBA8888），通过 HMIStateRules
  规则引擎根据主题/壁纸/充电状态选择暗色/白色/蓝色变体

  3.3 共享逻辑库

  两个渲染进程共享相同架构的数据处理层：

  库: LibClusterFunction
  路径: HMI/Lib/LibClusterFunction/
  作用: 主仪表的数据适配：Gauge、Warning、ADAS、IVI、行车电脑、HMI 状态
  ────────────────────────────────────────
  库: LibClusterFusa
  路径: HMI/Lib/LibClusterFusa/
  作用: FUSA 的数据适配：Telltale、Power、Theme

  每个库的核心组件：
  - LogicManager — 消息分发中枢，管理多个 FunctionManager
  - FunctionManager — 领域数据处理（每个功能一个）：FunctionGaugeInfoManager、FunctionAdasManager 等
  - HMIAdapter — 通过 DataProxyListener 监听 Middleware 数据变化
  - DataSource Plugin — Kanzi 插件，将 Middleware 数据写入 Kanzi 属性节点（双向桥接）

  3.4 数据绑定契约

  ClusterDataSource.xml 定义了 Middleware 与 Kanzi UI 之间的完整数据接口，涵盖
  Cluster、Menu、AVM、Gauge、ETC、DrivingComputer、Light、Warning、Charge、ADAS、Alarm 等命名空间。

  ---
  四、总结

  这个系统是一个双进程、功能安全分级的汽车仪表盘：

  - Middleware 负责所有业务逻辑——收发车辆信号、计算仪表值、管理电源状态、持久化设置、控制告警优先级
  - HMI 将 Middleware 的数据渲染为图形——主画面走 Kanzi 引擎实现丰富 UI，安全关键的警示灯走 QNX 原生 screen API
  保证可靠性
  - 模块间通过 FDBus IPC 松耦合通信，每个功能域独立为一个 Server
